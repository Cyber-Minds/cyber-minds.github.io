Risk #3 Mitigation

Security Review Checklist

Complete this checklist before any CTF challenge dataset, prompt, starter template, or fixture file is approved for release. Guards against accidental exposure of sensitive internals, answer-key artifacts, or real credentials.

Updated 2026-04-19 Owner: @ADGEEJU Required before every release
Progress
0 / 0
Secrets & Credentials Critical
No real API keys or tokens — confirm no API_KEY, AWS credentials, or personal access tokens appear anywhere in fixture files, starter code, or challenge prompts.
No real passwords or secrets — all passwords in sample logs, HTTP traces, or config files are clearly fictional (e.g., hunter2, changeme123).
No real SSH private keys — fixture files containing key material must use obviously fake RSA/EC key stubs that fail validation.
No production hostnames or IPs — all IP addresses must be RFC 5737 documentation ranges (192.0.2.x, 198.51.100.x) or clearly fictional.
Answer Keys & Spoilers Critical
No checker answer embedded in starter template — the starter code must not contain the exact string or pattern the checker looks for (e.g., don't pre-fill report.txt with a passing answer).
No answer-key files committed to repo — files like solution.sh, answer.txt, or expected-output.txt must live in a private branch or be excluded via .gitignore.
Challenge description does not over-hint — the objective text guides students toward the goal without revealing the exact command sequence or flag value.
Fixture Data & Logs High
Log fixtures are synthetic — sample.log, access logs, and auth logs are machine-generated with no real user names, emails, or internal hostnames.
HTTP traces are sanitised — any captured curl or proxy traces must have all Authorization, Cookie, and X-Api-Key headers redacted or replaced with [REDACTED].
Fixtures versioned in repo — all seed datasets live under a versioned path (e.g., fixtures/ctf-01/) and are not generated at runtime from a hidden source.
No PII in fixture data — no real names, email addresses, phone numbers, or real usernames appear in any challenge dataset.
Exploit Payloads & Dangerous Code High
No working exploit code published — challenges requiring vulnerability identification (e.g., CTF-07 IDOR) must use analysis tasks only; no functional exploit payload may appear in any file.
Shell commands are non-destructive — check scripts and starter code must not contain rm -rf, dd, curl | bash, or any command with irreversible side effects outside /workspace.
No network egress in check scripts — checker scripts must not make outbound HTTP requests; all validation is file-local to prevent data exfiltration vectors.
Release Gate Medium
CI smoke check passes — bash ctf01-check.sh (or equivalent) runs green against the seeded fixture workspace in the CI pipeline before merge.
Peer review completed — at least one team member other than the challenge owner has read through all fixture files and the checker script.
Roadmap leadership acknowledged — this checklist result is posted as a comment on the implementation sprint ticket before the challenge goes live.
